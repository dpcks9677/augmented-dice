# rules.md

일반적인 LLM 코딩 실수를 줄이기 위한 행동 지침입니다. 필요에 따라 프로젝트별 지침과 병합하세요.

**트레이드오프:** 이 지침은 속도보다 주의를 기울이는 것을 우선시합니다. 사소한 작업의 경우 판단력을 발휘하세요.

## 1. 코딩 전에 생각하기

**가정하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 표면화하세요.**

구현하기 전에:
- 가정을 명시적으로 진술하세요. 확실하지 않으면 질문하세요.
- 여러 해석이 존재하는 경우, 제시하세요 - 조용히 혼자 선택하지 마세요.
- 더 간단한 접근 방식이 있다면, 그렇다고 말하세요. 타당한 경우 반대 의견을 제시하세요.
- 명확하지 않은 부분이 있으면 멈추세요. 헷갈리는 부분이 무엇인지 명시하세요. 질문하세요.

## 2. 단순성 최우선

**문제를 해결하는 최소한의 코드. 추측적인 것은 배제합니다.**

- 요청받은 것 이상의 기능은 구현하지 마세요.
- 일회용 코드를 위한 추상화는 하지 마세요.
- 요청되지 않은 "유연성"이나 "구성 가능성"은 추가하지 마세요.
- 불가능한 시나리오에 대한 에러 처리는 하지 마세요.
- 200줄을 작성했는데 50줄로 줄일 수 있다면 다시 작성하세요.

스스로에게 물어보세요: "시니어 엔지니어가 이것을 너무 복잡하다고 할까?" 만약 그렇다면, 단순화하세요.

## 3. 외과적인 변경

**반드시 필요한 부분만 건드리세요. 자신이 어지른 곳만 정리하세요.**

기존 코드를 편집할 때:
- 인접한 코드, 주석 또는 포맷팅을 "개선"하지 마세요.
- 고장 나지 않은 것을 리팩토링하지 마세요.
- 자신이 선호하는 방식과 다르더라도 기존 스타일과 일치시키세요.
- 관련 없는 데드 코드를 발견하면, 삭제하지 말고 언급만 하세요.

변경 사항으로 인해 고립된 코드(orphan)가 발생하는 경우:
- '자신의' 변경으로 인해 사용되지 않게 된 임포트/변수/함수를 제거하세요.
- 요청받지 않는 한 기존 데드 코드를 제거하지 마세요.

테스트: 변경된 모든 줄은 사용자의 요청으로 직접 추적될 수 있어야 합니다.

## 4. 목표 주도적 실행

**성공 기준을 정의하세요. 검증될 때까지 반복하세요.**

작업을 검증 가능한 목표로 변환하세요:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성한 다음 통과하게 만들기"
- "버그 수정" → "버그를 재현하는 테스트를 작성한 다음 통과하게 만들기"
- "X 리팩토링" → "이전과 이후에 테스트가 통과하는지 확인하기"

다단계 작업의 경우, 간략한 계획을 명시하세요:
```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```

강력한 성공 기준은 독립적으로 루프를 돌 수 있게 해줍니다. 약한 기준("작동하게 만들기")은 끊임없는 확인이 필요합니다.

---

**이 지침이 효과적으로 작동한다면:** diff에 불필요한 변경이 줄어들고, 과도한 복잡성으로 인한 재작성이 줄어들며, 실수를 한 후가 아니라 구현 전에 명확한 질문을 하게 될 것입니다.
